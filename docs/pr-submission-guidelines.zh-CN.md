# PR 提交规范

[English version](pr-submission-guidelines.en-US.md)

## 1. 其他无效情形

- PR 描述空白或与实际代码变更严重不符。

- 引用了第三方库或框架，没有在 README 中列明依赖，且未说明原创功能部分。

- 复用了自己过去的代码片段，没在 PR 描述中注明来源。

## 2. PR（Pull Request）提交规范

### 2.1 基于 PR 开发

请基于 PR 添加新功能。

### 2.2 单一职责原则

每个 PR 只做一件事：

- 每个 PR 只实现或修改单一功能；
- 鼓励尽可能小、粒度尽可能细的 PR；
- 大功能应拆分成多个独立 PR 分步提交。

### 2.3 PR 内容要求

PR 标题与描述需清晰完整，内容包含：

#### 标题

一句话说明本 PR 新增/修改了什么。

#### 功能描述

说明该功能的作用与使用方式。

#### 实现思路

简要说明技术选型或核心实现逻辑。

#### 测试方式

如何验证该功能正常运行。

### 2.4 主分支要求

PR 合并后，主分支代码需保持可运行状态，评委在任意时间查看应能复现演示效果。
